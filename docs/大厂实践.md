# 大厂实践

## 大厂订单表分表策略

### 1. 主流方案：多维度组合
```text
维度                   策略                     适用场景
买家维度                user_id 取模分库         用户查询自己的订单列表（高频）
时间维度                按年月分表                订单按时间归档、历史数据清理
订单ID                 订单ID嵌入时间信息         根据订单号精准路由
```

### 2. 淘宝/阿里的实践
```text
分库：买家ID取模（如 % 16）→ 分散到16个库
分表：按年月命名（如 t_order_2024_01, t_order_2024_02）

订单号设计：19位，内含用户ID信息 + 时间信息
核心问题：同一个用户的订单分散在不同时间段的表中，但查询"我的订单"时需要聚合。
解决方案：
•异构索引表：订单数据按买家ID维度全复制一份到索引表
•双写机制：写入时同时写入主表和索引表
```

### 3. 美团/饿了么的实践
```text
分库：订单ID取模（如 % 32）
分表：按月分表（如 order_202401, order_202402）

优势：
•数据分布均匀，避免热点
•方便按时间归档历史数据
•支持冷热数据分离
```

### 4. 对比：取模 vs 按日期
```text
策略                      优点                          缺点
ID取模                    数据分布均匀，避免热点           扩容需要迁移数据
按日期                    便于归档，扩容简单               可能存在写入热点（新表压力大）
组合策略                   兼顾两者优点                   实现复杂度较高
```

## 订单号

```text
淘宝19位订单号的设计规则如下：
总长度：19位。
前13位：“根据时间戳和内部定义序列”。
后6位：“跟购买者ID相关的用户位”。


结构拆分
订单号：1234567890123456789（19位）
       └────┬────┘└──┬──┘
        前13位    后6位
       时间+序列   用户ID相关
各部分含义
位置          位数          含义                    作用
前13位        13位         时间戳 + 内部序列         保证全局唯一、有序
后6位         6位          用户ID后几位             用于分片路由、防重
```

```text
一、订单号设计的核心目标

先明确约束（这是设计的第一性原理）：
1. 全局唯一（强约束）
不能重复（分布式系统、多机房）
2. 趋势递增（重要）
便于：数据库索引（B+树友好）,分库分表（范围查询）,日志追踪
3. 高并发生成
QPS：几十万甚至百万级
4. 不暴露敏感信息
用户ID、订单量不能被轻易推测
5. 可扩展
支持未来扩容（机房、节点）
```

```text
对于阿里的19位纯数字订单号，在数据库中的存储，最主流和推荐的选择是使用 VARCHAR(19)。
核心原因在于，这串数字虽然长得像数值，但在实际业务中是作为一个代码（Code）来使用的，而非一个用于计算的数值（Number）。
你会想到用 BIGINT 来存，是因为它最大能存19位数字，刚好能放下。但在真实的工业级系统（比如阿里）的设计规范里，像订单号、身份证号这类字段，一律会存成字符串。


为什么是“19位”？
long 转字符串, long 最大：19位十进制
```
