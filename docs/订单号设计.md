# 订单号设计

## 一、订单号设计的核心目标

```text
先明确约束（这是设计的第一性原理）：

1. 全局唯一（强约束）
不能重复（分布式系统、多机房）

2. 趋势递增（重要）
便于：数据库索引（B+树友好）,分库分表（范围查询）,日志追踪

3. 高并发生成
QPS：几十万甚至百万级

4. 不暴露敏感信息
用户ID、订单量不能被轻易推测

5. 可扩展
支持未来扩容（机房、节点）
```

## 二、典型“阿里系订单号结构”（工程推测版）

```text
常见设计是一个64bit 或 128bit 的整型ID，再转成字符串（19位常见）

一个比较接近真实的结构是：
[时间戳] + [机器ID] + [序列号] + [业务位/混淆位]
```

```text
1. 时间戳（核心部分）

例如：
41 bit：毫秒时间戳（类似 Snowflake）

特点：
支持约 69 年
保证趋势递增
可用于：
    排序
    按时间分库

👉 注意：
很多人说“订单号里有时间”，这是对的，但：❗ 一般不会直接用“yyyyMMddHHmmss”这种明文

而是：
epoch offset（起始时间）
二进制压缩
```

```text
2. 机器ID（数据中心 + 节点）

例如：
10 bit：
  - 5 bit 数据中心
  - 5 bit 机器

作用：
避免多节点冲突
支持水平扩展
```

```text
3. 序列号（并发关键）

例如：
12 bit：每毫秒内自增

作用：
同一毫秒内生成多个订单
最大支持：
4096/ms/node
```

```text
4. “用户ID信息”到底是啥？

这是最容易被误解的点。

👉 实际上：
阿里不会直接把用户ID明文塞进订单号

原因：
安全问题（可推测用户规模）
隐私问题
热点问题（同用户高频下单）


更可能的做法是：
✅ 方案1：用户ID hash（部分位）
取 userId 的 hash 的低几位
作用：
参与分库分表路由
不可逆

例如：
userId % 1024 → 10bit


✅ 方案2：完全不放用户ID

很多系统（包括淘宝核心链路）：
👉 订单号 ≠ 路由键

而是：
分库键 = userId
订单号 = 全局唯一ID

这样,查询订单时：
先用 userId 定位库,再查订单号
```


## 关键工程优化点

你听到的是：“19位 = 用户ID + 时间”
更准确的说法应该是：“订单号通常包含时间趋势信息 + 分布式唯一性信息，可能包含用户ID的hash（而不是明文）”

```text
1. 时钟回拨问题

解决方案：
等待
使用备用workerId
或引入逻辑时钟


2. 热点问题

如果订单号是递增的：👉 会导致数据库写入热点

解决：
bit shuffle（位打散）
或：reverse timestamp


3. 分库分表路由 🎯🎯🎯💯💯💯

常见策略：
table = userId % N      (orderId最低bit位是用户相关的,就等价于按userId路由)
而不是：
orderId % N   ❌        (orderId最低bit位是递增序号的话,就等价于按orderId路由)

如果orderId的最低bit位是递增序号的话,新增orderId都会落到同一张表中. 如果orderId最低bit位是用户相关的话,就不会在分表这个级别出现数据倾斜了. 


4. 高可用
本地生成（不依赖DB）
无锁或CAS
多机房独立发号
```


## Leaf（美团开源）

```text
Leaf 是美团开源的一套分布式ID生成服务，本质上就是为了解决类似“订单号生成”这种高并发、全局唯一、低延迟的问题。
它在工程上比单纯的 Snowflake 更“工业化”，因为提供了两种模式 + 完整服务化方案。
```

```text
一、Leaf解决什么问题？

典型场景：
订单号（电商）
交易流水号（支付）
消息ID（MQ）
分布式主键（分库分表）

核心要求：
高并发 + 全局唯一 + 趋势递增 + 高可用
```

```text
二、Leaf整体架构

Leaf不是一个算法，而是一个服务系统：

        ┌────────────┐
        │  Client    │
        └─────┬──────┘
              │ HTTP / RPC
        ┌─────▼──────┐
        │  Leaf Server│
        ├────────────┤
        │ Segment模式 │
        │ Snowflake模式│
        └─────┬──────┘
              │
   ┌──────────▼──────────┐
   │ DB / ZK（依赖组件） │
   └─────────────────────┘
```
### Segment 模式（号段模式）
```text
Segment 模式（号段模式）
👉 本质：数据库分配“ID区间”

CREATE TABLE leaf_alloc (
    biz_tag     VARCHAR(128),
    max_id      BIGINT,
    step        INT,
    update_time TIMESTAMP
);

获取号段流程
1. Leaf从DB取一段ID：
   [100001 ~ 101000]
2. 缓存在内存中
3. 本地直接递增发号


双buffer优化（关键）
Leaf用了一个非常实用的优化：当前号段快用完时 → 异步加载下一个号段

优点
简单稳定
不依赖机器时钟
ID严格递增（非常重要）

缺点
依赖数据库
DB可能成为瓶颈（但可优化）
```

### Snowflake 模式

```text
Snowflake 模式
👉 本质：本地生成ID（类似Twitter Snowflake）

ID结构（标准64bit）
| 时间戳 | 数据中心 | 机器ID | 序列号 |

Leaf不是直接用 Snowflake，而是做了改进：
1. workerId 自动分配（基于 ZK）
启动时注册到 ZK,获取唯一 workerId,避免手动配置冲突

2. 时钟回拨保护
Leaf解决：
检测回拨,拒绝服务 or 等待,或切换workerId

"检测回拨"是通过在单机内存中维护一个变量lastTimestamp,每次生成 ID 时,通过临时比对系统时间戳和lastTimestamp来保证.
"切换workerId"是通过向zk申请一个不同的新的workerId,然后继续发号.(这样只能解决小范围了时钟回拨,否则就没办法满足订单号趋势递增的要求)
```

### 实际生产怎么选？
```text
如果是“订单号”（强推荐）
选：Segment模式

原因：
严格递增（对账/审计友好）
不依赖时钟（稳定）


如果是“通用ID / 日志ID”
选：Snowflake模式

原因：
性能极高
无DB依赖
```
